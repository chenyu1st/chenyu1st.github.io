运维手册——Centos7网络超时断开(后传)
发表于 2021-01-04| 更新于 2021-01-04|Linux
字数总计:2k|阅读时长: 6 分钟
| 阅读量:|评论数:

声明:这是我在大学毕业后进入第二家互联网公司学习的内容

背景

上篇已经介绍了，Centos7网卡中断，到最后都没有解决完成。

这个问题在最近又开始出现了，我这次换了一个思路来解决。

现状

现在的机器是一台12c 64g的服务器，性能比之前那台高不要太多，但是发现这台机器还是经常出现之前的报错：网卡中断，只能重启解决。

经过持续1个月的跟踪，我发现引发的问题的关键在nginx，无论我把nginx部署在哪台机器，都可能引发这个问题。

因为现象特别难复现，我专门蹲点（当有人喊xxx不能用的时候我就去那台机器调试）去尝试解决。

根据我部署的监控大盘数据分析，我得出了一个结论，这台机器的网络流量达到200-300MBS的时候就可能出现网卡中断的情况，但是机器并不是在峰值的时候就网卡中断，而是在峰值过后一段时间才出现这种情况。

问题分析

Nginx调优

我用的是我自己优化的一套配置方案，用了很久，我现在质疑我现在的优化是否存在问题，对比如下

  • 优化前
  • 优化后

我的调优参数主要添加了

  • CPU多核优化
  • epoll I/O多路复用模型
  • TCP优化
  • 链接参数优化
  • Gzip压缩

经过测试，我发现不调优的Nginx出现问题的频率更高（还是需要调优的）

Nginx并发和带宽瓶颈

经过查询资料发现

Nginx的CPU,内存和IO的利用方面是非常高效的

通过监控图也看出来来，基本占用不了很高的资源

下面看看并发和带宽的影响

并发

max_clients

这个参数没有出现在nginx的配置文件中

查资料发现

作为http web服务器角色的nginx能够处理的最大连接数就是最大客户端连接数。

最大客户端连接数计算公式为

作为反向代理服务器角色的nginx，用户发送请求到nginx，nginx发送请求到后端被代理服务器，后端服务器响应给nginx，nginx将内容返回给用户。nginx需要保持2个链接，所以最大的客户端连接数为

而目前这台机器是6c，主要功能也是反向代理，所以理论上它的并发是6*65535/2=196,605

而通过监控和日志，每秒钟并发也就50-300

远远低于nginx的最大并发

所以并发这块我认为没什么问题

带宽

我开始查询Nginx带宽限流策略，抓住几个主要流量的server进行调试

限流算法主要思想如下：

令牌桶算法

  • 令牌以固定速率产生，并缓存到令牌桶中；
  • 令牌桶放满时，多余的令牌被丢弃；
  • 请求要消耗等比例的令牌才能被处理；
  • 令牌不够时，请求被缓存。

漏桶算法

  • 水（请求）从上方倒入水桶，从水桶下方流出（被处理）；
  • 来不及流出的水存在水桶中（缓冲），以固定速率流出；
  • 水桶满后水溢出（丢弃）。
  • 这个算法的核心是：缓存请求、匀速处理、多余的请求直接丢弃。

从作用上来说，漏桶和令牌桶算法最明显的区别就是是否允许突发流量(burst)的处理，漏桶算法能够强行限制数据的实时传输（处理）速率，对突发流量不做额外处理；而令牌桶算法能够在限制数据的平均传输速率的同时允许某种程度的突发传输。

Nginx按请求速率限速模块使用的是漏桶算法，即能够强行保证请求的实时处理速度不会超过设置的阈值。

但是我们的场景应该需要令牌桶算法，不允许超过一个带宽值，但是又不能丢弃一个请求。

不过我测试配置了Nginx限流，多出的请求会返回503，影响用户体验。

操作系统带宽

没办法，看看能不能从操作系统的级别来限制带宽。

经过查询资料，通过限制网卡的速率来达到限流

限速百兆:
ethtool -s enp10s0 speed 100 autoneg off

恢复千兆:
ethtool -s enp10s0 speed 1000 autoneg on

执行限速百兆后，我明显感觉所有项目速度变慢

没办法，还是恢复吧，比起中断，一直慢速的体验会令人感到更差

解决问题

非根本解决法

查了一圈我表示以我目前的能力很难解决问题，先来个自动重启脚本吧

编写脚本

172.31.20.1是路由器

172.31.20.2是DNS服务器

如果这2个机器我都不能ping通即可说明这台机器已经和局域网失去了连接

并且可能存在某些异常情况导致服务器不能上网的时候，我要在启动这个脚本前留足缓冲时间，以便自己操作，时间设为600秒

vim /opt/ping-check.sh

#!/bin/bash
sleep 600
while true
do
   ping -c 3 -w 5 172.31.20.1
   if [[ $? != 0 ]];then
         ping -c 3 -w 5 172.31.20.2
         if [[ $? != 0 ]];then
              echo `date +"%Y-%m-%d %H:%M:%S"` >> /opt/restart.log
              reboot
         fi
   fi
   sleep 20
done

chmod +x /opt/ping-check.sh

将脚本加入system并开机自启

cd /usr/lib/systemd/system
vim ping-check.service

[Unit]
Description=check network
After=network.target remote-fs.target nss-lookup.target

[Service]
Type=forking
ExecStart=/bin/bash -c 'nohup /bin/bash  /opt/ping-check.sh  & echo $! > /opt/ping-check.pid'
ExecStop=/bin/bash -c 'cat /opt/ping-check.pid |xargs kill -9'

[Install]
WantedBy=multi-user.target

chmod 644 ping-check.service
systemctl start ping-check.service
systemctl enable ping-check.service
systemctl status ping-check.service

根本解决问题

编写完脚本后，感觉服务器不用自己重启了很方便，但是经过1个多星期的观察，平均服务器每天都要重启3-5次，在高峰的时候甚至刚刚启动就断网。

最后想了下，换网卡吧。

都是千兆网卡，我把这台机器的服务换到公司配置最高的服务器上，观察一段时间没问题。

那台机器还做了双网卡绑定，带宽更高。

总结

最终把这个问题解决了，持续跟踪了近2个月，收获不少，恶补了很多知识。

以后遇见这个问题直接换网卡或者采用多网卡绑定即可，软件层面太难啦。
